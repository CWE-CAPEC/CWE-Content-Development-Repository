NAME: Application State Type Corruption (ASTC) - Persistent Data Type Assumptions in PHP Applications

DESCRIPTION:

This weakness occurs when an application assumes that persisted data
structures will always conform to an expected type (e.g., objects) without
validating the retrieved state. In some cases, the application itself may
unintentionally mutate or convert the stored data into a different type
(e.g., arrays instead of objects) during normal processing. Subsequent code
that accesses properties or methods assumes the original type, leading to
persistent fatal errors across multiple modules. This can result in
authentication failures, system crashes, or other systemic failures.
Variations include any situation where unsafe assumptions about type
integrity are made across execution paths and persistent state mutation is
possible.

Additional Comments from Submitter:

Generalizability

This pattern is not language-specific. Although initially observed in PHP,
similar behavior has been confirmed across:
- Multiple languages (including Java). For example, in Axway SecureTransport,
the issue did not stem from the language itself (as it doesn't allow changing
data types), but from application-level handling (e.g., encoding/decoding
inconsistencies), where input was not transformed as expected and was instead
treated as a different data type, leading to the same class of failure
- Different frameworks and technology stacks
- Both web applications and enterprise softwares

The root cause is architectural, not related to dynamic typing. The corruption
originates from:
- Internal processing inconsistencies
- Encoding/decoding mismatches (e.g., authentication flows)
- Storage/retrieval transformations
- Application logic mutating persisted state (a key distinguishing
characteristic of this weakness)

Cross-Product Presence

This pattern has been identified across multiple real-world products, indicating
it is:
- Systematic and repeatable
- Present in shared design patterns
- Likely widespread across additional codebases

Impact Scope

This issue is often misclassified as DoS. However, it more accurately represents
persistent application state corruption or critical functionality disruption
(e.g., complete authentication failure where no users can log in)

Depending on context and execution flow, impacts include:
- Authentication and security control bypass (MFA, lockout, rate limiting). In
one observed case, malformed input led to authentication bypass and privilege
escalation due to corrupted state propagation
- Persistent state desynchronization
- Cross-module failure propagation
- Application-wide instability or collapse
- Potential account takeover scenarios

(Additional impact scenarios are detailed in the submitted research paper in the
mega link)

Key Distinction

The defining property is:

"The application mutates and persists structurally invalid state, and later
execution paths assume integrity without validation"

This combination of self-mutation + persistence + cross-module assumptions is
not covered by existing CWE entries

SUB.ATTACK - "Too attack-focused"

Description: The submission's name and/or description emphasizes the
attack, not the weakness. Either the submission's weakness is not
well-understood, or it is better described as an attack pattern, which may
be suitable for inclusion in CAPEC.

Resolution: An initial submission cannot progress to later phases unless it
also includes a clear weakness (i.e., is not subject to SUB.UNCLEAR).

Comments: Overall, the submission is too attack focused.  Much of the submission
details the attack and how to "corrupt" the state of the application.  CWE is
focused on what the root cause of the attack is.  Because the submission is
focused on the conditions to carry out the attack it is unclear what the
specific weakness being described is.  From the perspective of a researcher, it
is understandable that to arrive at a weakness, a new attack is discovered and
we need to work backwards.  The CWE team wants to understand exactly where the
submitter believes the weakness is.  There appears to be two distinct actions
that happen to cause this attack.

1. The application changes the type of a resource based on unvalidated
user input 2. The application accesses the changed data type with a
previous data type that is now incompatible (type confusion - covered
by CWE-843: Access of Resource Using Incompatible Type)

We are hoping that the submitter can identify which of these scenarios they
believe the proposed weakness to be.


Response: The intended weakness is not the specific method used to
introduce malformed state, but the failure to enforce structural
invariants on persisted application state.

Specifically, the weakness occurs when:

- The application persists state that does not conform to the expected structure or type

- That state is later treated as valid canonical data across execution boundaries

- No validation or normalization is performed when rehydrating or
accessing this persisted state The resulting data type inconsistency
(e.g., object vs array) is one manifestation of this condition, but
not the defining characteristic.

The defining weakness is:

Failure to validate the structure or type of persisted application
state before use across execution contexts.

This represents a failure to validate data when crossing a persistence
boundary, rather than a failure limited to a single execution point.

Response: Acknowledged. The weakness is clarified as a failure to

SUB.OVERLAP - "Duplicate/Overlap"

Description: The submission identifies a weakness, but it is a duplicate of
(or partially overlaps) existing weaknesses or other submissions,
indicating problems with the submission or associated CWE entries.

Resolution: The submission cannot progress to later phases until (1) the
duplicate/overlap is understood and (2) it is clear how to resolve this
duplicate/overlap.

Comments: This submission heavily overlaps with CWE-843: Access of Resource
Using Incompatible Type ('Type Confusion').  In the reference to this
submission, the submitter addresses this overlap by stating the difference is
this submission involves "self-mutation of persisted state".  The claim is that
this is different to type confusion because the type and access was originally
correct, although the application mutates the type such that the access is no
longer correct.  The CWE team disagrees that there is a difference here. CWE-843
should cover the situation where the product accesses a resource using an
incompatible type.  We would like the submitter to explain more about why the
submission poses a unique weakness from CWE-843.  Is this issue with the
phrasing from CWE-843: "incompatible with the original type", because the
application has mutated the type to be something other than the original?

Response: This submission overlaps with CWE-843: Access of Resource
Using Incompatible Type, but differs in the origin and propagation of
the failure.

In CWE-843, the issue is primarily at the point of access, where a
resource is used with an incompatible type.

In this submission, the issue arises earlier in the lifecycle:

- The application persists or stores state that does not conform to
  the expected structure or type

- That state is later assumed to be valid without revalidation or
  normalization

- This leads to downstream type mismatches during usage

The distinguishing aspect is the implicit trust boundary between
persistent storage and application logic.

In this case, the application treats persisted data as inherently
trustworthy and structurally valid, without revalidation when crossing
this boundary. This allows structurally inconsistent state to
propagate across execution contexts.

This aspect is not explicitly emphasized in CWE-843, which focuses on
incompatible type usage but does not highlight persistence boundaries
or state rehydration assumptions.

If this is not considered sufficient for a new entry, this behavior
could be incorporated into CWE-843 by explicitly extending its scope
to include:

- Persistent storage of data that does not conform to the expected
  structure or type

- Lack of validation during state rehydration

- Cross-execution reuse of inconsistent or invalid state

CWE-843 assumes the incompatible type exists at point-of-use

This submission introduces a precondition failure before access
occurs, where persisted state becomes part of the trusted execution
model without revalidation across boundaries.
