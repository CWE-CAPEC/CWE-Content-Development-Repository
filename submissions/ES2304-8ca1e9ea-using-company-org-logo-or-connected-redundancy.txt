NAME: Using company/org logo or connected with them items in redundancy situations

DESCRIPTION:

From time to time some orgs/companies are using logo or connected with them
items in redundancy situations which can provide to unwanted actions like
be an 'authorized' by them for bad actors (illegal actions). Example:
company as an added feature let to all users make a QR code for site or
something else with logo company or company connected - feature should just
make a QR code which works correct (from user point of view and usability),
but adding a company's logo by default can be read as redundant and
unwanted action, because they don't know how users (so also bad actors)
will exploit this feature. In short words bad actors can exploits that QR
code with company's logo in really easy way - some victims will be remember
the malicious results with company's logo. There can be many examples of
redundant adding of org/company connected items - exploiting of
bussiness/marketing logic where intentions of some feature were good, but
possibility of exploits by bad actors are large.

It represents an unnecessary attack vector or an avoidable opportunity for

to focus on a single example (although similar cases may occur elsewhere,

Illustrative example - potential risk: Depending on which browser is used,
a user can often access a feature via RMB -> "Create QR code for this
page."

The purpose of this feature is to generate and display a correct QR code -
that's its intended, legitimate use. However, some browsers automatically
add their logo or branding element to the generated QR code, most likely
for marketing or recognition purposes.

While this is fine in benign use cases, the situation changes when a
malicious actor takes advantage of it - e.g., generating a QR code that
leads to malicious content. The difference is that the browser's logo,
added by default, unintentionally lends trust and credibility to the QR
code.

Of course, a malicious actor could add such branding manually, but in this
case, it's automatic and default behavior, which lowers the effort
threshold for abuse. In my view, since the resulting QR code's destination
or intent is not verified or controlled, it would be safer to avoid adding
organizational branding by default, to prevent potential misuse and
reputational risk.

The QR code scenario is just an example meant to illustrate a broader
point: similar concerns could appear in other, non-QR-related contexts -
particularly where secure-by-design principles should apply. In that sense,
I believe it's a topic worth consideration.

In a slightly different context, this could resemble a kind of "server-side
spoofing", where an external actor can influence a generated output,
affecting security indirectly. However, in this case, the issue relates
more to the software's design and behavior than to web mechanisms
themselves.

I'd be happy to clarify further if needed, and I'm genuinely interested in
your thoughts on this topic.
